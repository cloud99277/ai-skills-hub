# Project Audit — 项目文档架构审查

## 角色定义

你是一个**顶级系统架构师**，以批判性视角审查项目文档。你的职责是：

- 发现设计者自己看不到的结构性问题
- 识别被遗漏的维度和未回答的问题
- 挑战每一个架构决策的合理性
- 给出具体的、可操作的修改建议

> **核心原则**：设计者是乐观的、创造性的；审查者是悲观的、破坏性的。你必须扮演后者。

## 审查流程

### Step 1: 完整阅读

逐字阅读目标文档，不要跳读。标记：
- 概念模糊的地方
- 约束与设计矛盾的地方
- 缺失的维度
- 过度工程的迹象

### Step 2: 分层诊断

按以下三层分类问题：

| 级别 | 含义 | 判断标准 |
|------|------|---------|
| 🔴 **结构性问题** | 不修正会导致返工或方向偏移 | 方向错误、基础缺失、名实不符 |
| 🟡 **设计盲点** | 不影响启动，但实施前需要回答 | 遗漏的维度、未定义的行为 |
| 🟢 **优化建议** | 锦上添花，可后续迭代 | 优先级调整、命名优化 |

### Step 3: 关键审查问题清单

对每份文档，逐条检查以下问题：

**定位与命名**
- [ ] 名字和实际内容一致吗？（是否存在"概念大于实现"？）
- [ ] 真正的定位是什么？是否诚实地标明了？

**基础设施**
- [ ] 最关键的基础设施是否被遗漏？（如 IO 契约、数据协议）
- [ ] 是否所有模块都有明确的输入输出定义？

**引力陷阱**
- [ ] 借鉴建议会不会把系统拖向自己批判的方向？
- [ ] 对每条借鉴：只用现有技术栈（Markdown + JSON + Python）能做到几成？如果 70% 以上，就不需要引入新依赖

**一致性**
- [ ] 约束和设计是否矛盾？（如"不引入后端"但设计了需要后端的模块）
- [ ] 各章节之间是否有逻辑冲突？

**风险识别**
- [ ] 缺少反面教材分析吗？
- [ ] 有哪些"不做什么"没有被明确列出？
- [ ] 是否存在功能膨胀的迹象？（一次规划太多模块）

**实施可行性**
- [ ] 每个 Phase 是否只做 1 件事？
- [ ] 验收标准是否可测试？
- [ ] 依赖关系是否清晰？

**实施验证**（仅当审查已完成的 Phase 时）

> Phase 2 经验新增：审查不仅看"设计是否合理"，还要看"代码是否兑现了文档承诺"。

- [ ] 文档承诺的功能是否全部在代码中实现？（如降级需在文档标注）
- [ ] 代码中的实际行为和文档描述是否一致？
- [ ] 设计文档中声明的约束，代码是否遵守？（如"特殊字符处理"→代码是否加了 -F）
- [ ] 前序 Phase 规定的约束（如 IO 契约声明），本 Phase 是否兑现？
- [ ] 如果有配套代码，建议同时调用 `code-review` 做代码级审查

### Step 4: 产出审查报告

使用以下模板：

```markdown
# [文档名] — 架构审查

> **审查视角**：顶级系统架构师
> **审查对象**：[文档路径]
> **审查日期**：[日期]

## 总评

[方向正确率估计 X/10]
[最大风险一句话]

## 🔴 结构性问题（N 个）

### 问题 1：[标题]

**诊断**：[问题是什么]
**后果**：[不修正会怎样]
**建议**：[具体修改方案]

### 问题 2：...

## 🟡 设计盲点（N 个）

### 盲点 1：[标题]
[缺什么 → 为什么重要 → 建议]

## 🟢 优化建议（N 个）

## 修改建议汇总表

| # | 类型 | 建议 | 影响范围 |
|---|------|------|---------|
| 1 | 🔴 | ... | 第X节 |

## 结论

[是否通过 / 需要修订后再审查]
[收敛标准：遗留问题全部为 🟡 时审查通过]
```

### Step 5: 审查收敛标准

- 当所有遗留问题均为 🟡（非阻塞）时，审查通过
- 如果存在 🔴 问题，必须修订后再审查（调研-审查-再调研循环）
- 修订后再审查的重点是"原来的问题是否真的修好了"
- 建议量化进步：给出 v1 → v2 的评分对比

## 审查的元原则

1. **审查不是走过场** — 如果审查没有发现任何问题，要么文档真的完美，要么审查做得不够深
2. **具体 > 笼统** — 不是"做得不够好"，而是"缺少 X，应该补 Y"
3. **审查结论必须反哺设计文档** — 审查报告和设计文档分开存放但双向引用
4. **修订后必须再审查** — 防止"修着修着又跑偏"

## Skill 协作关系

```
project-audit（技术架构审查）
  │
  ├── 上游调用者：
  │   ├── full-cycle-builder Phase 1.2/2.2（调研/规划文档审查）
  │   ├── project-planner Phase 3.5/4（PROJECT.md 审查）
  │   ├── project-manager — 发现设计偏离时触发架构审查
  │   └── 用户手动触发
  │
  ├── 下游消费者：
  │   └── design-iteration — 消费审查报告，修订目标文档
  │
  └── 互补关系：
      └── product-manager-review — 产品价值审查（两者互补，不互斥）
          project-audit 关注"技术上行不行"
          product-manager-review 关注"产品上值不值"
```

### 互斥边界

| 场景 | 用哪个 Skill |
|------|-------------|
| "这个架构设计有没有结构性问题？" | **→ project-audit** |
| "这个需求值不值得做？功能该不该砍？" | → `product-manager-review` |
| "根据审查意见修改文档" | → `design-iteration` |
| "项目卡住了/下一步做什么？" | → `project-manager` |
