# State, Writeback, and Checks

## 状态模型

任何结论都要先归类，再决定写到哪里。

使用四种状态：

- `confirmed`
  - 已被用户明确确认，且短期内不太会变化
- `tentative`
  - 当前倾向成立，但用户还没有完全确认
- `open`
  - 仍待补充或待回答的问题
- `conflict`
  - 不同文档或不同轮回答之间存在冲突

规则：

- `confirmed` 可以进入确认摘要正文
- `tentative` 只能留在详细档案或单独的 open items
- `open` 不进入确认摘要
- `conflict` 必须先标注再解决，不要强行合并

## 写详细档案时

详细档案应该保留：

- 用户原话的关键信息
- 你的结构化归纳
- 哪些是 `confirmed` / `tentative` / `open` / `conflict`
- 还值得继续追问什么

详细档案允许保留推理过程与过渡判断。

## 写确认摘要时

确认摘要只保留：

- 已经稳定成立的结论
- 当前清晰的路线、偏好、边界、终局定义

确认摘要不要混入：

- “暂定”
- “可能”
- “第 5 个还没想好”
- “剩余多少以后再定”

如果确实需要保留未定项，把它们放到独立的 `Open Items` 小节，而不是混进 confirmed 主体。

## 写基础资料文件时

只有在信息天然属于这些文件时才更新：

- `profile.md`
- `preferences.md`
- `personal-operating-manual.md`

不要为了“同步”而到处重复写同一段内容。

## 一致性检查

每轮更新后，至少检查这几件事：

1. 确认摘要是否只包含稳定结论
2. 详细档案上半部分是否还在声称“缺失”，但后文其实已经补齐
3. `tentative` / `open` 是否混进 confirmed 主体
4. 同一条结论是否在不同文件中互相打架
5. 路径变化后，关键文件是否仍可被正确发现

如果发现问题：

- 先修边界
- 再修 wording
- 最后再考虑扩写
